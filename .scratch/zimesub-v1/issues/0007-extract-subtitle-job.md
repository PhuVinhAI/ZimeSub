## What to build

User clicks "Trích xuất sub" on an Episode that has a `selected_subtitle_track_id`. A `Job` of kind `ExtractSubtitle` is enqueued and runs in a background tokio task. Progress (parsed from `mkvextract` stderr by `progress_parsers::parse_mkvextract`) streams to the Episode row as a percentage. On success, `<basename>.eng.ass` exists in the EpisodeFolder and EpisodeState advances to `Extracted`. If the picked track was SRT, it is converted to ASS during/after extract so the on-disk file is always `.eng.ass`.

This slice ships a minimal queue: serial execution, no UI other than per-Episode inline progress bars. Full tiered concurrency + bottom status bar arrive in 0008.

## Acceptance criteria

- [x] "Trích xuất sub" button enabled only on Episodes with `selected_subtitle_track_id` set; tooltip on disabled state explains.
- [x] Clicking enqueues a `Job { kind: ExtractSubtitle, episode_id, mkv_track_id }`. Minimal queue runs jobs serially in a tokio task.

## Implementation notes

The slice splits along the existing thin-command pattern: three new pure Rust modules (`progress_parsers`, `ass_ops`, `episode_state`) hold the testable logic, a fourth new module (`job_queue`) owns the background tokio worker + cancellation primitives + post-process step, three new Tauri commands wire them up (`episode_inspect_artifacts`, `extract_subtitle_start`, `extract_subtitle_cancel`), and the frontend gains one new TS api file (`extract.ts`), one new Solid store (`jobs.ts`), one new design-system primitive (`ProgressBar`), two new modal components (`ExtractConfirmModal`, `ExtractErrorModal`), and a substantial `EpisodeRow` rewrite that drives all of the above. No new Tauri capabilities or new Rust dependencies were required — `tokio` (process + io-util + macros + sync + rt features) was already added in slice 0001 / 0006 and `uuid` is generated frontend-side; `core:event:default` is already present in `capabilities/default.json`.

### Pure modules

`progress_parsers::parse_mkvextract(line) -> Option<ProgressUpdate>` is the AC's named parser. mkvextract emits progress as a fresh `Progress: N%` line per percent (no carriage-return-driven TUI redraws), so the parser is a prefix + suffix strip with whitespace tolerance for the occasional trailing `\r` on Windows pipes. The returned `ProgressUpdate { ratio: f32, hint: String }` carries a `[0, 1]`-clamped fraction the frontend renders directly and a short human-readable label (`"35%"`) for the row's "x%" hint. Out-of-range percentages (`>100`) and non-numeric matches (`abc%`, `12.5%`, `-5%`) return `None` so a future mkvextract format change can never paint the bar past 100% or as nonsense. Seven unit tests cover the zero / mid / full happy paths, whitespace + CR tolerance, the various non-progress line shapes that must be silently dropped, the out-of-range rejection, and the non-numeric rejection.

`ass_ops::srt_to_ass(srt) -> String` is the AC's "convert SRT in place" implementation. The header it prepends is a minimal `[Script Info]` + `[V4+ Styles]` (one `Default` style: Arial 48 white-on-transparent with a 2 px outline / 2 px shadow / center-bottom alignment) + `[Events]` triplet so mpv, VLC, and Aegisub all load the file without complaint. The per-block converter accepts both standard SRT (numeric index → timing → text lines) and the looser index-less shape some non-Windows exporters produce (timing on the first line), tolerates either `,` or `.` as the millisecond decimal separator (rare but legitimate), and truncates milliseconds → centiseconds rather than rounding (rounding can push a cue past the next one's start and produce mid-line flicker; truncation always stays ≤ original so adjacent cues never collide). Multi-line dialogue joins with the ASS `\N` escape; the trailing `Text` field passes commas verbatim because ASS dialogue lines treat the last column as a literal until end-of-line. Malformed individual blocks (missing timing, unparseable timestamps, `mm >= 60`, `ss >= 60`) are silently skipped so a partial source still produces a loadable file rather than an all-or-nothing failure. Eleven unit tests cover empty input, single-block round-trip, multi-line `\N` joining, the index-less shape, CRLF input scrubbing, the dot decimal separator, multi-block input, mid-stream malformed block recovery, every timestamp validation branch, ms→cs truncation at the 999 / 990 / 001 boundaries, and the comma-in-text passthrough.

`episode_state::inspect_artifacts(folder, basename) -> EpisodeArtifacts` is the disk half of the PRD's `derive_state` pseudocode. It returns a typed snapshot with four booleans (`has_extracted_sub`, `has_translation_draft`, `has_translated_sub`, `has_render`) so the wire shape is stable from slice 0007 onward — only the first is computed today, but the others ship as always-`false` placeholders so the frontend JobsStore + EpisodeRow consumers don't need a follow-up rewrite when later slices flip them on. The function performs one `Path::is_file` syscall per artefact and never mutates the folder; pure-ish but for those reads, which is exactly what the PRD's "pure function over disk + queue snapshot" framing calls for. Five unit tests using a tempdir cover the empty folder, the `.eng.ass` flip, the directory-not-file defensive case (someone manually creates a *folder* named like the artefact), the exact-vs-substring basename matching (sibling Episodes in the same Project folder don't bleed across rows), and the real-world `[Erai-raws] Show - 01 [1080p][HEVC][hash]` basename shape.

### Background job runner

`job_queue::JobQueue` owns a single tokio worker task that drains an `mpsc::UnboundedSender<ExtractSubtitleSpec>` and runs one job at a time. The `Arc<Mutex<HashMap<String, JobControl>>>` map keyed by `job_id` holds the per-job cancellation primitives — an `Arc<Notify>` for the in-flight wake-up signal AND an `Arc<AtomicBool>` cancel flag — so a cancel that arrives between `enqueue` and the worker picking the job up is still observed (the worker checks the flag at the top of `run_extract_subtitle` and short-circuits to a `cancelled: true` `job-done` event). Cancellation is idempotent: calling `cancel(id)` on a job that's already finished is a no-op. The worker removes the control entry from the map after each job completes, so memory doesn't grow with session length.

For each job the worker emits three Tauri events: `job-started` (right before mkvextract spawns; flips the row from `queued` → `running` so the progress bar appears even before the first percentage crosses the wire), `job-progress` (one per parsed `Progress: N%` line), and `job-done` (success / failure / cancellation with the full captured stderr buffer for the failure modal). The stderr accumulator is held in an `Arc<tokio::sync::Mutex<String>>` shared between the stderr-reader task and the supervisor so the final payload contains every line the child emitted — including non-progress warnings and the post-mortem error messages mkvextract prints on failure.

mkvextract is invoked with `tokio::process::Command::new(mkvextract_path).args(["tracks", source_mkv, "<track_id>:<output.ass>"]).current_dir(episode_folder).kill_on_drop(true)` and the Windows `CREATE_NO_WINDOW` flag (mirroring the convention `install.rs` and `tooling.rs` already use) so no console window flashes in front of the user during the extract. `stdin` is `Stdio::null()` because mkvextract never reads from stdin and leaving it inherited would block on user input in the rare interactive-prompt case. `stdout` is still piped but drained-and-discarded because mkvextract emits very little on it (banner + "Extracting track …") — none of which the frontend currently surfaces.

The cancel path runs `child.start_kill()` and `child.wait()` to reap the OS process before the cleanup pass. `cleanup_partial_output` walks the EpisodeFolder for `<basename>.eng.ass` AND `<basename>.eng.srt` (the latter is reserved for a future change that might introduce an intermediate file — currently mkvextract writes straight to `.eng.ass`) and deletes any that exist. Without this an aborted job would leave a half-written `.ass` that `derive_state` would later misidentify as a real extract.

Post-extract SRT detection runs after a clean `Ok(0)` exit. `looks_like_srt` inspects the first non-empty line of the on-disk file: if it's `[Script Info]` it's ASS (no conversion); if it's `WEBVTT` magic it's something else we don't touch; if it's a numeric block index and the second line contains the SRT ` --> ` arrow it's SRT, and if the first line itself contains the arrow it's the index-less SRT variant. On a positive match, `ass_ops::srt_to_ass` runs and the result is atomically rewritten in place via the same `tmp + rename` pattern `project_store` and `settings_store` use for their manifests — so a crash during the SRT-to-ASS rewrite never leaves a half-flushed `.ass` on disk. A post-process failure (read error, write error) downgrades the otherwise-successful outcome into a failure with a Vietnamese error string so the row's red "Lỗi extract" badge surfaces the rewrite problem; the cleanup pass then removes the half-converted file.

Nine unit tests cover the SRT-detection heuristic (standard block, index-less block, CRLF input, ASS-header rejection, WEBVTT-magic rejection, empty / garbage rejection), the output-path convention (`<basename>.eng.ass`), the cleanup pass (removes the two known candidates without touching unrelated files in the same folder), and the atomic write (overwrite of an existing file).

### Tauri commands & state

`commands::AppState` gains a `OnceLock<Arc<JobQueue>>` field — the worker is lazily instantiated on the first job command because `JobQueue::new` needs an `AppHandle`, which only Tauri command handlers see. `OnceLock` keeps the init thread-safe and avoids spawning the worker task on apps that never reach the pipeline (e.g. an Onboarding-only session). Three new commands wrap the job and disk operations:

  * `episode_inspect_artifacts(folder, episode_id) -> EpisodeArtifactsView` — read-only disk probe the frontend calls on project open / project switch (once per Episode), after each `job-done` (to flip the badge to "Đã extract" the moment mkvextract finishes), and on the overwrite-confirm path (to decide whether to surface the modal at all). The returned view echoes back the EpisodeFolder name as `output_basename` so the frontend can render the resolved artefact path (`<basename>.eng.ass`) in tooltips and the future "Open in Explorer" affordance without a second round-trip through `resolve_episode_targets`.
  * `extract_subtitle_start(job_id, folder, episode_id)` — resolves the episode targets, validates the selection (rejects with the Vietnamese "Chưa chọn subtitle track cho Episode này" when `selected_subtitle_track_id` is `None`), reads `mkvextract` from `settings.tool_paths` (rejects with "Chưa phát hiện đường dẫn mkvextract" when the cache is empty), and dispatches the spec to the queue. `job_id` is generated frontend-side (uuid via `crypto.randomUUID()`) so log + progress + done events can be correlated with the originating click, mirroring the winget install flow's `installId` convention.
  * `extract_subtitle_cancel(job_id)` — idempotent cancellation by id, surfaces no error when the id is unknown (already finished, already cancelled).

### Frontend wiring

`src/api/extract.ts` mirrors the four Rust event shapes 1:1 and exposes the three `invoke` bindings plus three `listen`-based subscribe helpers (`onJobStarted`, `onJobProgress`, `onJobDone`) — same pattern `src/api/install.ts` uses for the winget streaming flow. The TS types match the Rust `#[derive(Serialize)]` outputs exactly: `JobStartedEvent { job_id, episode_id }`, `JobProgressEvent { job_id, episode_id, ratio, hint }`, `JobDoneEvent { job_id, episode_id, success, cancelled, exit_code, error, stderr }`.

`src/stores/jobs.ts` is the slice 0007 JobsStore. It holds two per-Episode maps (`jobs: Record<episodeId, EpisodeJobState>` for the live extract-job phase + progress + sticky stderr buffer, `artifacts: Record<episodeId, EpisodeArtifactState>` for the disk artefacts cache) and a session-only `dontAskOverwrite: Record<episodeId, boolean>` set for the "Không hỏi lại cho Episode này" checkbox. `ExtractJobPhase` is `'queued' | 'running' | 'success' | 'failed' | 'cancelled'` — queued before the backend ack, running on `job-started`, then one of the three terminal states on `job-done`. `success` / `cancelled` clear after a 600 ms tick so the row settles into its disk-only "Đã extract" appearance; `failed` is sticky so the red "Lỗi extract" badge persists until the user dismisses the error modal.

The store is autonomous — it doesn't import the projects store. AppShell bridges between them via a `createEffect(on(() => ({ folder, ids: projectsStore.active.episodes.map(e => e.id) }), snapshot => setActiveProject(snapshot.folder, snapshot.ids)))` watcher that fires whenever the active project changes (open, switch, or Episode append). `setActiveProject` drops every per-Episode signal from the previous project, then re-inspects the disk for every Episode in the new one so each row's badge boots with the correct "Trống" vs "Đã extract" derivation. Selectors (`jobStateFor`, `artifactStateFor`, `shouldConfirmOverwrite`) return non-null records so callers don't have to defensively null-check — the missing-entry case maps to the pristine "no job has run / no artefacts cached" snapshot.

Event subscriptions install once per app instance via `ensureJobSubscriptions()` (called from `AppShell.onMount`), guarded by a module-level promise + handle pair so Solid's dev double-mount and HMR never produce duplicate listeners. Each handler filters by `(jobId === current.jobId)` before applying state so events from a previous job for the same Episode are silently dropped — without this, a `job-done` event from a stale subscription could clobber a fresh job's state. `handleDone` schedules a `refreshArtifactsForEpisode` async refresh so the row's badge flips to "Đã extract" the moment mkvextract finishes (the badge update is independent of the 600 ms job-state cleanup tick — the progress bar disappears as soon as the phase flips, and the badge updates when the artefact cache fills).

### Design-system + view changes

`src/design-system/ProgressBar.tsx` is a new flat-style determinate progress bar primitive — 2 px border track, accent fill, `h-2` so it sits comfortably inside the 56 px Episode row. `ratio` is `[0, 1]`-clamped before rendering so a parser bug shouldn't paint the bar past 100% or as a sliver of negative width; ARIA `progressbar` role + min/max/now values + `aria-valuetext` (defaults to the rounded percentage) make the bar announce its state to screen readers without callers needing to wire the fallback themselves.

`src/views/project/ExtractConfirmModal.tsx` is the AC-mandated "Ghi đè bản extract hiện có?" confirm. Modal body shows the Episode folder name in mono (so the user knows which Episode they're about to overwrite even if multiple rows fired the modal in quick succession), a Vietnamese explanation that mentions the on-disk `.eng.ass` file and the consequence of overwriting it, and a checkbox bound to a `dontAsk` signal that resets on every open (so a previous "checked" state doesn't bleed across Episode rows). Footer: secondary "Hủy" button + primary "Ghi đè" button. The checkbox emits `rememberDontAsk` through `onConfirm(rememberDontAsk)`; ProjectView calls `rememberDontAskOverwrite(episodeId)` on the JobsStore before kicking the actual extract. Session-only memory — cross-session persistence would be a schema change for a convenience flag and the AC doesn't ask for it.

`src/views/project/ExtractErrorModal.tsx` is the AC-mandated "click-through to view the captured stderr" failure modal. Opened by clicking the red "Lỗi extract" badge on an Episode row; renders the buffered stderr text via the existing `TerminalLog` primitive (slice 0003) with `stream: 'stderr'` for the warn tint. The summary line above the log reads "mkvextract thất bại (exit N) · <error message>" with the exit code and the optional backend-supplied Vietnamese error string (spawn failure, post-extract conversion failure, etc.) folded into one paragraph. Dismissing the modal calls `clearJobState(episodeId)` so the row collapses back to its disk-only appearance (the underlying artefact is still missing, so the badge falls back to "Trống" via the artefact cache).

`src/views/project/ProjectView.tsx` is the substantial change. The right-hand cluster on each `EpisodeRow` is now four slots: language tag (`StatusBadge tone="neutral"`, slice 0006), middle state slot (one of progress bar / "Đang chờ" / "Lỗi extract" clickable badge / "Đã extract" / "Trống"), primary action button (one of "Chọn track" / "Trích xuất sub" / "Hủy" / "Thử lại" — state machine driven by `hasSelection`, `isInFlight`, `isFailed`), and the optional "Đổi track" link (only when the row has a track selected AND no job is in flight, so the link isn't competing for attention with the cancel button). When a job is running, the middle slot becomes a 160 px wide flex row with the progress bar plus a `w-10 text-right` hint label so the percentage stays right-aligned across all rows. When no track is selected, the AC's "Trích xuất sub button enabled only on Episodes with `selected_subtitle_track_id` set; tooltip on disabled state explains" requirement is satisfied by rendering the button in its disabled state with `title="Chọn subtitle track trước"` next to the (still functional) "Chọn track" affordance — clicking either one moves the user forward.

`StateSlot` and `ActionButton` are kept as separate sub-components inside `ProjectView.tsx` so the row template doesn't drown in nested `<Show>` branches. The precedence order is explicit: running → queued → failed → has-extracted → empty for the state slot, and no-selection → in-flight → failed → idle for the action button. `EpisodeRow` reads the live job snapshot and artefact cache directly from `@stores/jobs` (no prop drilling for those), but takes callback props for actions that need to surface modals owned by `ProjectView` (`onExtract`, `onShowError`, `onCancelExtract`, `onPickTrack`) so the modal mounting stays at the ProjectView root.

The `handleExtractRequest` entry point in `ProjectView` is what implements the AC's "Nếu `<basename>.eng.ass` already exists, a confirm modal appears" rule: it consults `shouldConfirmOverwrite(episodeId)` on the JobsStore (true iff `artifacts.hasExtractedSub && !dontAskOverwrite[id]`) and routes between opening the modal or directly enqueueing. The modal's `onConfirm(rememberDontAsk)` callback flips the don't-ask memory when checked, closes the modal, and then enqueues — keeping the "confirmed then started" sequence linear so a race where the user re-opens the modal mid-confirm can't double-enqueue.

`AppShell.tsx` adds `void ensureJobSubscriptions()` to `onMount` alongside `bootstrapTools()` so backend job events are listened for from the moment the app boots (no events are fired before a project opens, but Onboarding can complete before the user opens a project so the subscription must be live regardless). The active-project bridge `createEffect` reads `projectsStore.active` + `projectsStore.activeFolder` reactively and emits `setActiveProject(folder, ids)` to the JobsStore — tracked by `(folder, episode IDs)` so re-opening the same project or appending Episodes both trigger the artefact-cache re-sync.
